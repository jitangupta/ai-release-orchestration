# Jira Schema Definition
| Field Name             | Data Type     | Description                                                                              | Purpose in RAG Context Generation                                                              |
|------------------------|---------------|------------------------------------------------------------------------------------------|------------------------------------------------------------------------------------------------|
| issue_key              | STRING        | The unique Jira ID (e.g., JIRA-9013).                                                    | Primary Retrieval Key. Used to fetch the full context after a match in the Release Content.    |
| issue_type             | ENUM          | The kind of work (Bug, Story, Spike, Technical Debt).                                    | Crucial for the LLM to interpret the nature of the change and its urgency.                     |
| summary                | STRING        | The short title of the ticket (e.g., "Proration logic fails on cross-currency upgrade"). | Provides immediate, high-level context for the final recommendation sentence.                  |
| status                 | ENUM          | Current status (e.g., Resolved, Done).                                                   | Ensures only completed work is used for upgrade recommendations.                               |
| linked_feature_id      | STRING        | The ID of the feature affected (e.g., F3_SubscriptionMgmt).                              | Links the ticket content to the features used by the tenant.                                   |
| priority               | ENUM          | Internal engineering criticality (Highest, High, Medium, Low).                           | Provides the LLM with quantitative data on the internal risk of the issue.                     |
| description            | TEXT (LONG)   | The original full text detailing the problem or requested feature.                       | Primary RAG Content (Problem): Used to describe what went wrong or what was requested.         |
| resolution_details     | TEXT (LONG)   | Details on how the issue was fixed, including root cause analysis.                       | Primary RAG Content (Solution): Used to describe how the tenant will benefit from the upgrade. |
| impacted_modules       | ARRAY<STRING> | Technical components affected (e.g., api-gateway, db-migration-f3, helm-chart).          | Allows for granular risk analysis related to technical deployment complexity.                  |
| customer_context       | OBJECT        | Details about the tenant who reported/requested the issue.                               | Prioritization Factor: Crucial for prioritizing upgrades for key tenants.                      |
| $\quad$ customer_id    | STRING/NULL   | The ID of the specific tenant/customer.                                                  |                                                                                                |
| $\quad$ is_key_account | BOOLEAN       | Flag if the customer is a key or strategic account.                                      |                                                                                                |
| fix_version            | STRING        | The application version where the fix/feature was released (e.g., v1.5.0).               | Explicitly tracks when the change was available.                                               |
| release_notes_excerpt  | TEXT (MEDIUM) | A clean, human-readable summary intended for customer release notes.                     | Used to generate polished, non-technical reasoning for the final recommendation.               |

# Tenant Schema Definition

| Field Name                   | Data Type     | Description                                                                                          | Rationale for AI Recommendation Engine                                                                                           |
| ---------------------------- | ------------- | ---------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| tenant_id                    | STRING        | Unique identifier for the tenant (e.g., T-CORP-401).                                                 | Primary key for data retrieval.                                                                                                  |
| tenant_name                  | STRING        | Human-readable name (e.g., "Acme Corp").                                                             | For human-operator readability in the final report.                                                                              |
| deployment_type              | STRING        | How the tenant's app is deployed (shared-namespace, dedicated-namespace, dedicated-cluster).         | Affects the complexity and risk of the Helm/GitOps deployment.                                                                   |
| current_version              | STRING        | The currently deployed application version (e.g., v1.4.2).                                           | Crucial baseline for determining if an upgrade is a minor patch or a major jump.                                                 |
| risk_tolerance               | ENUM          | The tenant's documented willingness to adopt new releases (LOW, MEDIUM, HIGH).                       | Direct input for the SHOULD/CAN SKIP decision. LOW tolerance means higher stability requirements.                                |
| active_features              | ARRAY<STRING> | A list of feature IDs currently enabled/used by the tenant (e.g., ["F1_Proration", "F3_TrialMgmt"]). | If a new release fixes a critical bug in a feature they use, the upgrade necessity increases dramatically.                       |
| daily_usage_score            | INTEGER       | A normalized score (1-100) reflecting overall daily usage volume/API calls.                          | Higher usage means higher risk for a broken release, thus increasing scrutiny.                                                   |
| key_metrics_last_30_days     | OBJECT        | Key performance indicators relevant to the SaaS product.                                             | Allows AI to cross-reference usage patterns with bug reports (e.g., "If $F1$ usage is > 90%, prioritize fix").                   |
| $\\quad$ total_subscriptions | INTEGER       |                                                                                                      |                                                                                                                                  |
| $\\quad$ active_api_users    | INTEGER       |                                                                                                      |                                                                                                                                  |
| $\\quad$ failed_jobs_percent | FLOAT         |                                                                                                      |                                                                                                                                  |
| contact_slack_channel        | STRING        | The primary channel for technical alerts (e.g., #acme-support-ops).                                  | Useful for the future ability to integrate with the operator workflow (e.g., "AI posts upgrade recommendation to this channel"). |


## Rationale for Key Tenant Profile Columns
### 1. active_features (e.g., ["F1_Proration", "F2_Webhooks"])
| Rationale               | AI Function                                                                                                                                                                                                |
| ----------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Necessity Filter        | This is the single most important column for the RAG retrieval. The AI needs to know, "Does this tenant use the feature that was just updated/fixed?"                                                      |
| Upgrade Risk Assessment | If the new release contains a critical fix for F1_Proration, and the tenant uses F1_Proration, the recommendation changes from "CAN SKIP" to "MUST Upgrade" (to mitigate a known, active risk).            |
| Skip Justification      | If the new release only affects Feature F5, and the tenant does not use F5, the AI can confidently recommend "CAN SKIP" and provide the reasoning: "Tenant is not exposed to the changes in this release." |

### 2. daily_usage_score (e.g., 78)
| Rationale                 | AI Function                                                                                                                                                                                                                                                                                 |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Operational Impact Weight | This acts as a simple, high-level weighting factor for risk. A tenant with a high usage score (e.g., 95) represents a higher potential business impact if the upgrade fails compared to a low-usage tenant (e.g., 10).                                                                      |
| Prioritization            | In a large batch of tenants recommended for upgrade, the AI (or the human operator) can use this score to prioritize deployments. The top 10 highest-score tenants might be upgraded first after the initial canary validation, or conversely, held back until maximal stability is proven. |

### 3. key_metrics_last_30_days (Structured Usage Data)
| Rationale             | AI Function                                                                                                                                                                                                                                                                                                                           |
|-----------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Granular Risk Context | While daily_usage_score is general, these metrics allow for specific, quantitative cross-referencing with known bugs.                                                                                                                                                                                                                 |
| Example Scenario:     | If a bug ticket (retrieved from the Vector DB) states: "High failure rate for tenants processing > 10,000 subscriptions," the AI can check the tenant's total_subscriptions metric. If the tenant is above that threshold, the necessity to fix the bug via upgrade becomes critical and specific reasoning can be generated.         |
| Failed Jobs Percent:  | This is a real-time health check. If a tenant is already experiencing stability issues (failed_jobs_percent is high), the AI should factor this in: either "MUST Upgrade" if the fix addresses the root cause, or "SHOULD NOT Upgrade yet" if the upgrade introduces high complexity and the current environment is already volatile. |

### 4. contact_slack_channel (Operational Data)
| Rationale                          | AI Function                                                                                                                                                                                                                                                                         |
|------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Actionable Output/Function Calling | This is included with an eye toward the full AI-Orchestrated vision. The final goal isn't just a recommendation; it's a trigger for action.                                                                                                                                         |
| Future Integration                 | This field is a perfect input for a Function Call or Agent action. For instance, after the GitOps PR is merged, the system could automatically send a notification to the designated Slack channel: "Upgrade for Alpha Solutions Inc. to v1.5.0 approved and deployment initiated." |
